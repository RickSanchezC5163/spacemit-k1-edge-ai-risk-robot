## 一、赛题硬性要求逐项审计

### 1. 依托主控芯片内置 NPU 或高性能 CPU，在边缘侧部署轻量化 AI 模型

| 要求 | 当前状态 | 差距 |
|------|---------|------|
| 使用 K1 芯片 NPU 或 CPU | 所有代码在伴生 ARM 计算机运行，未触及 RISC-V 核心 | **严重缺失** |
| 部署轻量化 AI 模型 | 红色检测使用 HSV 规则，非 AI 模型 | **严重缺失** |
| 模型量化/剪枝 | 无模型可量化 | **缺失** |

**赛题明确说**："重点考察模型在嵌入式端的量化、剪枝与部署能力"。这是核心得分点，当前为空白。

### 2. 严禁单纯依赖云端 API，必须包含本地推理环节

| 要求 | 当前状态 | 差距 |
|------|---------|------|
| 不依赖云 API | ✅ 全本地 | 满足 |
| 包含本地推理 | LLM-A 是模板替换，非推理 | **未满足** |

"本地推理"意味着模型在本地运行前向传播，而非规则匹配或模板填充。

### 3. 设计需具有明确的实际意义

| 要求 | 当前状态 | 差距 |
|------|---------|------|
| 实际意义 | ✅ 管廊巡检场景 | 满足 |
| 目标识别 | HSV 规则检测，非目标识别系统 | 弱 |

### 4. 完成从感知输入、模型推理到执行输出的完整链路

| 要求 | 当前状态 | 差距 |
|------|---------|------|
| 感知输入 | ✅ N10P LiDAR + D435 | 满足 |
| **模型推理** | ❌ HSV 规则 + 模板替换 | **链路断裂** |
| 执行输出 | ✅ C30D 底盘 + 5-DOF 臂 | 满足 |

### 5. 重点考察模型在嵌入式端的量化、剪枝与部署能力

| 要求 | 当前状态 | 差距 |
|------|---------|------|
| 量化 | 无 | **缺失** |
| 剪枝 | 无 | **缺失** |

## 五、8 天倒排计划

### 最高优先级：补上"模型推理"缺口（这几天最重要的事）

**Day 1-2 (7/1-7/2)：准备风险数据集 + 部署轻量检测模型**

```
目标: 用 YOLOv8-nano 或 MobileNet-SSD 替换 HSV 红色规则

步骤:
1. 收集/打印 3-5 类风险图片（裂缝、锈蚀、堵塞、积水、变形）
   每类 10-20 张，贴在纸板上
2. 用 D435 拍摄这些风险卡片，标注 bbox
3. 在 x86 上训练 YOLOv8n（~3M 参数），导出 ONNX
4. 转换为 K1 NPU 支持的格式（需确认 K1 NPU SDK）
5. 在 K1 上运行推理，替代 d435_red_rule_detector.py
```

如果 NPU SDK 不可用或时间不够，退而求其次：
```
方案 B: 在 K1 ARM CPU 上跑 ONNX Runtime + YOLOv8n
  - CPU 推理，帧率会低（可能 1-3 FPS）但至少是"模型推理"
  - 可以展示量化前后对比（int8 vs fp32）
```

**Day 3 (7/3)：部署本地轻量 LLM**

```
目标: 替换模板 LLM-A 为真实本地推理

方案 A: llama.cpp + Qwen2-0.5B 或 Phi-3-mini
  - 在 K1 ARM CPU 上运行
  - Token 生成速度估计: 2-8 tokens/s
  - 量化: Q4_K_M 或 Q8_0

方案 B: llama.cpp + TinyLlama-1.1B
  - 更小更快，更适合嵌入式展示

方案 C (保底): 如果 ARM 跑不动
  - 用服务器跑 llama.cpp 但强调"可在 RISC-V 部署"
  - 展示架构图和量化方案
```

**Day 4 (7/4)：Gazebo 仿真环境 + 风险场景**

```
1. 在服务器 Ubuntu 上安装 ROS 2 Jazzy + Gazebo Harmonic
2. 构建管廊 world: T 型分叉 + 窄道 0.8m
3. 放置障碍物 SDF 模型
4. 贴风险纹理到墙壁（打印数据集中的图片作为纹理）
5. 验证 /scan, /camera/color/image_raw 正常
```

**Day 5 (7/5)：Gazebo 中跑通完整 pipeline**

```
Gazebo:
  guarded motion → D435 capture → 模型检测风险 → 落图 → 机械臂响应 → LLM 报告
```

**Day 6 (7/6)：性能测量 + 量化对比**

```
1. 测量并记录:
   - YOLOv8n 推理帧率 (FPS)
   - LLM token/s, TTFT
   - 内存占用
   - CPU 占用

2. 量化对比:
   - FP32 vs INT8 的帧率/精度对比表
   - 模型大小对比（量化前 vs 量化后）
```

**Day 7 (7/7)：录制演示视频 + 写材料**

```
1. 录制 Gazebo 仿真视频（画中画）
2. 录制真机视频（如果 K1 上能跑模型）
3. 写竞赛报告/PPT
4. 整理项目文档
```

**Day 8 (7/8)：最终收口检查**


**你现在的项目在"工程完整度"上很好——底盘/雷达/相机/机械臂/安全/协议全部验证通过。但在赛题核心得分点"AI 模型推理"上是空白——没有 NPU 部署、没有量化模型、没有本地 LLM 推理。未来 8 天必须把重心从"多走几步"转移到"让模型跑起来"。** 最小可行方案：YOLOv8n ONNX + llama.cpp Qwen2-0.5B，在 K1 ARM CPU 上跑通，记录性能数据。
