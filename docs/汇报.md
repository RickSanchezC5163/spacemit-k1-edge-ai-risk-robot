# 组会汇报稿

这段时间我主要完成了小车从硬件驱动到边缘 AI 演示链路的基础打通，现在项目已经从“单个模块调通”推进到“整车稳定运行和演示链路打磨”阶段，准备在接下来几天补齐资料，上交素材。

作品主要是基于N10P雷达、D435深度相机的对于外部环境的感知,通过N10P雷达进行稳定的二维建图,结合YOLO模型识别地图上的风险点,并结合深度相机的雷达数据,准确地标定出地图上某个位置存在的风险。同时如果有堵塞的识别,最后通过机械臂进行排障的一个对于受限环境内的整个风险的排查系统。
整车的自主操作实现基于gazebo仿真和ROS2 Humble的Nav2导航栈,通过RRT算法进行自主建图和路径规划,结合雷达和深度相机的感知数据,实现对于受限环境内的自主探索和风险排查。

其中，雷达30Hz，车底盘20Hz，D435 30Hz。YoloV8n 10Hz，Qwen-0.5b模型每秒20 tokens。
YoloV8n Qwen-0.5b模型都是本地部署的，完全不依赖云端推理。基于官方提供的Ep核加速，相比于全cpu有明显的性能提升。

结合题目要求,需要本地部署大模型,此处考虑到了对于风险识别的汇总应用。大模型主要是通过对于本地资料,比如说图片、文字,我们的风险汇报的汇总,结合具体的地图环境，最后呈现给用户一个稳定的受限区域内地图样貌，地图上的各个风险判定,以及我们车自主完成的一些风险排查动作的汇报。

## 1. 底层硬件联调

目前 K1 Muse Pi Pro 已经重新刷入 Bianbu LXQT v2.3.3 比赛固件，并完成了 ROS2 Humble 环境适配。

底盘使用 C30D Tank 固件，N10P 雷达、D435 深度相机、PWM 补光灯、总线舵机和本地 LLM 服务都已经完成基础接入测试。

其中底盘一开始存在停车拖尾和 odom 严重不准的问题。后面通过修改 STM32 固件里的编码器参数，把 Tank 车型从错误的 500 线光电编码器参数改成 GMR 编码器参数，odom 和实际运动的一致性明显改善。

## 2. 安全底盘和建图链路

目前已经跑通：

```text
N10P /scan
+ C30D /odom
+ odom -> base_footprint TF
+ slam_toolbox /map
```

同时加了一层安全控制，不让 Nav2 或手动速度直接裸连底盘。

当前结构是：

```text
/input_cmd_vel 或 Nav2 /cmd_vel_raw
-> scan_safety_guard_node
-> /cmd_vel_guarded
-> Python Tank base
-> C30D
```

这样可以做低速人工建图、小目标 Nav2 测试，并且前方有障碍时可以触发安全停车。

目前还没有宣称完整自动探索，只是完成了“人工/受控低速建图 + 安全停车”的 MVP。
